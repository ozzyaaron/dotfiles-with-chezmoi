# Ralph Template Reference

Reference templates for every file `/ralph init` and `/ralph plan` produce. **These are starting points, not canonical outputs.** For each project, evaluate every section against the project's actual needs — language base image, lockfile, secret surface, network endpoints, MCP servers — and adapt accordingly. The manifest (`ralph.config.yaml`) is where most per-project variation lives; the Dockerfile, launcher, and firewall script accept that variation through build args and config reads, so you rarely need to edit them.

When a template includes a `{{PLACEHOLDER}}`, replace it with a project-specific value. When a template uses a concrete value (e.g., `node:20-bookworm`), confirm it matches the project's stack and swap if not.

## Security model

The loop's tool surface is the security boundary. Inside the container, the in-container `claude` is invoked as:

```
claude -p \
  --bare \
  --tools "" \
  --strict-mcp-config \
  --mcp-config /etc/ralph/mcp.json \
  --allowed-tools "mcp__ralph_tools__*,<additional MCPs from manifest>" \
  --permission-mode default \
  --output-format stream-json --verbose \
  < PROMPT.md
```

What each flag does:
- `--bare` — disables hooks, plugin sync, auto-memory, keychain reads, and CLAUDE.md auto-discovery. Auth must come from `ANTHROPIC_API_KEY` (env) or an explicit settings file. Strips all the implicit context that could leak host state into the loop.
- `--tools ""` — disables every built-in tool (Bash, Edit, Write, Read, WebFetch, WebSearch, Task, TodoWrite, etc.). The loop literally cannot call these.
- `--strict-mcp-config --mcp-config /etc/ralph/mcp.json` — ignores `.mcp.json`, user MCP config, project MCP config; only the file passed here is loaded. Prevents leakage of host-side MCP servers (Gmail, Drive, Calendar, etc.) into the loop.
- `--allowed-tools` — pre-approves the listed MCP tools so no headless prompt is needed.
- `--permission-mode default` — failure-closed; anything not pre-approved is denied.
- No `--dangerously-skip-permissions` — the boundary is the tool surface, not permission-bypass.

The `ralph-tools` MCP server is the audited interface to the rest of the system. Everything else (Bash, network beyond Anthropic, host filesystem, secrets, git remotes) is structurally unavailable.

## ralph.config.yaml

The per-project manifest. Lives at `$PROJECT/ralph.config.yaml`, generated by `/ralph plan`, consumed by `bin/ralph`. Loop reads it as ro-mounted `/etc/ralph/config.yaml` — workspace copy is informational; runtime authority is the mount.

```yaml
schema_version: 1                                        # launcher refuses if mismatched

# What this project is and how to build/test it. Stack-agnostic.
stack:
  name: "ruby-rails"                                     # informational; "ruby-rails" / "node" / "python" / "go" / etc.
  base_image: "ruby:4.0.2"                               # Docker base image for the loop's container
                                                         # if omitted, launcher derives from stack.name + version files
  lint_cmd: "bundle exec rubocop -a"                     # exact command run by run_lint MCP tool
  test_cmd: "bundle exec rspec"                          # exact command run by run_tests MCP tool
  install_cmd: "bundle install --frozen"                 # exact command run by install_dependencies MCP tool

# MCP servers to bake into the image and load in the loop. Empty default.
# Each entry is a pinned package with sha256 checksum verified at build time.
mcp_servers: []
# Example entries:
#   - { name: "context7",         package: "@upstash/context7-mcp@1.0.5",                  sha256: "abc..." }
#   - { name: "postgres-readonly", package: "@modelcontextprotocol/server-postgres@0.6.3", sha256: "def...", env: { DSN: "${RALPH_POSTGRES_DSN}" } }

# Network policy applied by init-firewall.sh.
network:
  allowed_domains:                                       # default: only api.anthropic.com
    - "api.anthropic.com"
  registries: []                                         # rubygems.org / registry.npmjs.org / pypi.org if install_cmd runs during loop
  dns_resolver: "1.1.1.1"                                # pinned; UDP/53 only to this address

# Files in the workspace that must be masked or refused inside the sandbox.
# Refused by MCP tools for read AND write. Materialized as masked copies in
# .ralph/sandbox-workspace/ at `bin/ralph up`.
sensitive_paths:
  - ".env"
  - ".env.*"
  - "config/master.key"
  - "config/credentials*.yml.enc"
  - ".git/config"
  - "ralph.config.yaml"                                  # the manifest itself
  - "bin/ralph"                                          # the vendored launcher
  - "bin/ralph.d/**"                                     # vendored Dockerfile, mask-env.js, ralph-tools/, etc.
  - ".ralph/**"                                          # masked dir, audit, sandbox-workspace itself
  - "PROMPT.md"                                          # loop instructions
  - "IMPLEMENTATIONPLAN.md"                              # mutated only via mark_* tools
  - "specs/**"                                           # planner-authored; loop reads only

# Env var NAMES the running code references. Values are masked deterministically
# by `bin/ralph mask-env` (no LLM in the loop) and injected into the test/lint
# subprocess env. The planner enumerates names; it never reads real values.
masked_env_vars: []
# Example: ["DATABASE_URL", "REDIS_URL", "STRIPE_SECRET_KEY"]

# git_commit identity (since .git/config is masked).
git_commit:
  author_name: "Ralph Loop"
  author_email: "ralph@local"
  skip_hooks: false                                      # true to bypass project pre-commit hooks (defense vs. malicious dep hooks)

# Host exposure — every entry is opt-in.
host_exposure:
  docker_internal: false                                 # false → no --add-host=host.docker.internal:host-gateway
  db_port: null                                          # null or e.g. 5432; if set, the firewall allows host_ip:db_port

# Additional MCP tool ids to pre-approve. mcp__ralph_tools__* is always prepended.
loop_allowed_tools: []
# Example: ["mcp__context7__*", "mcp__postgres-readonly__*"]

# Per-iteration caps to bound a runaway or compromised loop.
quotas:
  max_tool_calls: 200
  max_bytes_read: 50000000                               # 50 MB
  max_bytes_written: 5000000                             # 5 MB
```

### Validation rules

The launcher and `bin/ralph audit` enforce:
- `schema_version` must equal 1. Future schema bumps require explicit `bin/ralph migrate-config`.
- `stack.lint_cmd`, `test_cmd`, `install_cmd` are exact commands. No shell metacharacter expansion in the MCP server — they are passed as `argv` to `spawn`, not to `bash -c`.
- `sensitive_paths` always includes the entries listed above (the launcher injects them if missing). User cannot opt out of masking the launcher itself, the manifest itself, the plan, or `.ralph/`.
- `network.allowed_domains` always includes `api.anthropic.com` (re-injected if dropped).
- `mcp_servers[].sha256` must match the package's tarball checksum at build time, or the build aborts.
- `loop_allowed_tools` glob patterns are restricted to `mcp__*__*` form. Built-in tool ids (Bash, Edit, etc.) are silently dropped.
- `quotas.*` are required; absence is treated as zero (i.e., loop refuses to start).

## Dockerfile

Stack-agnostic. The planner picks the language base image and `install_cmd` via the manifest; the Dockerfile itself doesn't know about Ruby vs Node vs Python.

```dockerfile
# syntax=docker/dockerfile:1.6
ARG BASE_IMAGE={{BASE_IMAGE}}                            # e.g. "ruby:4.0.2" or "node:20-bookworm" or "python:3.12-slim"
FROM ${BASE_IMAGE}

ARG TZ
ENV TZ="$TZ"

# ---- System packages: minimum for the loop ----
# No gh / vim / nano / fzf / man-db / zsh — those are interactive-shell tools.
# Bash being denied at the tool surface means the LLM has no use for them.
RUN apt-get update && apt-get install -y --no-install-recommends \
      git \
      ca-certificates \
      curl \
      jq \
      libcap2-bin \
  && apt-get clean && rm -rf /var/lib/apt/lists/*

# ---- Two users: ralph (LLM-controlled) and ralph-tools (MCP server) ----
ARG RALPH_UID=1000
ARG RALPH_GID=1000
ARG TOOLS_UID=1001
ARG TOOLS_GID=1001

RUN groupadd --gid ${RALPH_GID} ralph && \
    useradd --uid ${RALPH_UID} --gid ${RALPH_GID} --create-home --shell /bin/sh ralph && \
    groupadd --gid ${TOOLS_GID} ralph-tools && \
    useradd --uid ${TOOLS_UID} --gid ${TOOLS_GID} --create-home --shell /bin/sh ralph-tools

# No NOPASSWD sudo. No sudoers entry at all. Firewall runs in a separate
# privileged init container; the workload container has no sudo need.

ENV DEVCONTAINER=true
WORKDIR /workspace

# ---- Node 20 (for Claude Code + MCP servers) ----
# Pinned. No `latest`. No curl|bash for installation scripts.
ARG NODE_VERSION=20.18.0
RUN ARCH=$(dpkg --print-architecture) && \
    case "$ARCH" in \
      arm64) NODE_ARCH=arm64 ;; \
      amd64) NODE_ARCH=x64 ;; \
      *) echo "Unsupported arch: $ARCH" >&2; exit 1 ;; \
    esac && \
    curl -fsSL "https://nodejs.org/dist/v${NODE_VERSION}/node-v${NODE_VERSION}-linux-${NODE_ARCH}.tar.xz" -o /tmp/node.tar.xz && \
    mkdir -p /opt/node && \
    tar -xJf /tmp/node.tar.xz -C /opt/node --strip-components=1 && \
    rm /tmp/node.tar.xz && \
    ln -s /opt/node/bin/node /usr/local/bin/node && \
    ln -s /opt/node/bin/npm /usr/local/bin/npm && \
    ln -s /opt/node/bin/npx /usr/local/bin/npx

# ---- Claude Code (pinned) ----
ARG CLAUDE_CODE_VERSION={{CLAUDE_CODE_VERSION}}          # e.g. "2.1.141"
RUN npm install -g @anthropic-ai/claude-code@${CLAUDE_CODE_VERSION}

# ---- ralph-tools MCP server ----
# The launcher stages the MCP server source into the build context as
# `ralph-tools/` (today: from `~/.claude/ralph/ralph-tools/`; in a published
# distribution this could be a pinned + checksummed release tarball fetched
# at build time instead — adapt for your install).
# Root-owned and write-stripped so the `ralph` user cannot modify it at runtime.
COPY ralph-tools/ /opt/ralph-tools/
RUN cd /opt/ralph-tools && \
    npm install --omit=dev --no-audit --no-fund && \
    chown -R root:root /opt/ralph-tools && \
    find /opt/ralph-tools -type f -exec chmod a-w {} + && \
    find /opt/ralph-tools -type d -exec chmod a-w {} + && \
    chmod 0755 /opt/ralph-tools/bin/ralph-tools

# ---- Project-declared MCP servers (pinned + checksummed via build args) ----
# Generated by the launcher from ralph.config.yaml.mcp_servers and passed as build args.
ARG MCP_INSTALL_SCRIPT={{MCP_INSTALL_SCRIPT}}
RUN if [ -n "${MCP_INSTALL_SCRIPT}" ]; then \
      echo "${MCP_INSTALL_SCRIPT}" | sh -e ; \
    fi

# ---- Default claude settings (immutable; bind-mounted ro at runtime) ----
RUN mkdir -p /etc/claude-defaults/hooks
COPY claude-settings.json /etc/claude-defaults/settings.json
# No PreToolUse hooks (Bash is disabled; rtk rewrite is dead in loop mode).

# ---- Firewall script (used by the init container, not the workload container) ----
COPY init-firewall.sh /usr/local/bin/init-firewall.sh
RUN chmod 0755 /usr/local/bin/init-firewall.sh

# ---- Drop to ralph user. No sudoers. ----
USER ralph

ENV SHELL=/bin/sh
ENV NPM_CONFIG_PREFIX=/home/ralph/.npm-global
ENV PATH=/home/ralph/.npm-global/bin:/usr/local/bin:/usr/bin:/bin
```

Build-arg notes for the launcher:
- `BASE_IMAGE` comes from `stack.base_image` in the manifest. If the planner left it empty, the launcher falls back to a stack-aware default (`ruby:<version-from-.ruby-version>`, `node:20-bookworm`, `python:3.12-slim`, etc.). Prefer setting `stack.base_image` explicitly — it pins the build and removes inference.
- `MCP_INSTALL_SCRIPT` is a generated shell snippet of `npm install` or `pip install` calls, one per entry in `mcp_servers`, each verified by sha256 before installing.

## Firewall

Runs in a privileged init container that sets up iptables in the shared network namespace, writes the state file, and exits. Workload container has no `NET_ADMIN` and inherits the netns.

```bash
#!/bin/bash
# init-firewall.sh — runs in privileged init container.
# Reads /etc/ralph/config.yaml, applies iptables policy, writes state to
# /etc/ralph/iptables-state.txt (consumed by `bin/ralph audit`).

set -euo pipefail
IFS=$'\n\t'

CONFIG=/etc/ralph/config.yaml
STATE=/etc/ralph/iptables-state.txt

# Parse allowed_domains and dns_resolver from the manifest with jq.
# Requires yq to convert YAML → JSON, or a sidecar pre-process step.
ALLOWED_DOMAINS=$(yq -o=json '.network.allowed_domains // []' "$CONFIG" | jq -r '.[]')
DNS_RESOLVER=$(yq -r '.network.dns_resolver // "1.1.1.1"' "$CONFIG")
DB_PORT=$(yq -r '.host_exposure.db_port // ""' "$CONFIG")

# Preserve Docker's internal DNS NAT rules before flushing.
DOCKER_DNS_RULES=$(iptables-save -t nat | grep "127\.0\.0\.11" || true)

iptables -F
iptables -X
iptables -t nat -F
iptables -t nat -X
iptables -t mangle -F
iptables -t mangle -X
ipset destroy allowed-domains 2>/dev/null || true

if [ -n "$DOCKER_DNS_RULES" ]; then
  iptables -t nat -N DOCKER_OUTPUT 2>/dev/null || true
  iptables -t nat -N DOCKER_POSTROUTING 2>/dev/null || true
  echo "$DOCKER_DNS_RULES" | xargs -L 1 iptables -t nat
fi

# Loopback always allowed.
iptables -A INPUT  -i lo -j ACCEPT
iptables -A OUTPUT -o lo -j ACCEPT

# DNS — only to the configured resolver, UDP/53.
iptables -A OUTPUT -p udp --dport 53 -d "$DNS_RESOLVER" -j ACCEPT
iptables -A INPUT  -p udp --sport 53 -s "$DNS_RESOLVER" -j ACCEPT

# No outbound SSH (TCP/22), no outbound TCP except to allowed_domains.
# No broad UDP, no broad anything.

ipset create allowed-domains hash:net

# Resolve each allowed domain and add its IPs.
for domain in $ALLOWED_DOMAINS; do
  ips=$(dig +noall +answer A "$domain" "@$DNS_RESOLVER" | awk '$4 == "A" {print $5}')
  if [ -z "$ips" ]; then
    echo "WARNING: failed to resolve $domain" >&2
    continue
  fi
  while read -r ip; do
    [[ "$ip" =~ ^[0-9]{1,3}\.[0-9]{1,3}\.[0-9]{1,3}\.[0-9]{1,3}$ ]] || continue
    ipset add -exist allowed-domains "$ip"
  done <<<"$ips"
done

# Optional: host-DB port (e.g., postgres on the host).
if [ -n "$DB_PORT" ]; then
  HOST_IP=$(ip route | awk '/default/ {print $3; exit}')
  if [ -n "$HOST_IP" ]; then
    iptables -A OUTPUT -p tcp -d "$HOST_IP" --dport "$DB_PORT" -j ACCEPT
    iptables -A INPUT  -p tcp -s "$HOST_IP" --sport "$DB_PORT" -m state --state ESTABLISHED -j ACCEPT
  fi
fi

iptables -P INPUT DROP
iptables -P FORWARD DROP
iptables -P OUTPUT DROP

iptables -A INPUT  -m state --state ESTABLISHED,RELATED -j ACCEPT
iptables -A OUTPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
iptables -A OUTPUT -m set --match-set allowed-domains dst -j ACCEPT
iptables -A OUTPUT -j REJECT --reject-with icmp-admin-prohibited

# Write the state file for `bin/ralph audit` to consume.
{
  echo "# Generated by init-firewall.sh at $(date -u +%FT%TZ)"
  echo "# dns_resolver=$DNS_RESOLVER"
  echo "# db_port=${DB_PORT:-none}"
  echo "## allowed_domains:"
  for d in $ALLOWED_DOMAINS; do echo "$d"; done
  echo "## ipset:"
  ipset list allowed-domains
  echo "## iptables:"
  iptables-save
} > "$STATE"

# Self-verify.
if curl --connect-timeout 5 https://example.com >/dev/null 2>&1; then
  echo "ERROR: example.com reachable — firewall not enforcing" >&2
  exit 1
fi
if ! curl --connect-timeout 5 https://api.anthropic.com >/dev/null 2>&1; then
  echo "ERROR: api.anthropic.com unreachable — firewall over-restrictive" >&2
  exit 1
fi
echo "Firewall configured: $(echo "$ALLOWED_DOMAINS" | wc -w) domains allowed."
```

Periodic re-resolution: a cron in the workload container (read-only, but cron can run since it doesn't need writability) or a sidecar timer container runs the resolve step every 10 minutes to refresh ipset entries for CDN-rotating domains.

## MCP config

Generated at `bin/ralph up` and bind-mounted ro at `/etc/ralph/mcp.json`. Constructed from `ralph.config.yaml > mcp_servers`. Workspace `.mcp.json` (if any) is ignored due to `--strict-mcp-config`.

```json
{
  "mcpServers": {
    "ralph_tools": {
      "type": "stdio",
      "command": "/opt/ralph-tools/bin/ralph-tools",
      "args": ["--config", "/etc/ralph/config.yaml"]
    }
  }
}
```

When `mcp_servers` in the manifest is non-empty, additional entries appended programmatically. Example with `context7`:

```json
{
  "mcpServers": {
    "ralph_tools": {
      "type": "stdio",
      "command": "/opt/ralph-tools/bin/ralph-tools",
      "args": ["--config", "/etc/ralph/config.yaml"]
    },
    "context7": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@upstash/context7-mcp"]
    }
  }
}
```

The `ralph_tools` MCP server is always present and runs as the `ralph-tools` system user (uid 1001), not as the `ralph` user that runs claude.

## Claude settings

Bind-mounted ro at `/home/ralph/.claude/settings.json` from `/etc/claude-defaults/settings.json`. **No PreToolUse hooks** (no Bash to rewrite). No `Stop` hooks. Minimal.

```json
{
  "$schema": "https://json.schemastore.org/claude-code-settings.json",
  "permissions": {
    "deny": ["Bash", "WebFetch", "WebSearch", "Edit", "Write", "Read", "Glob", "Grep", "Task", "TodoWrite", "NotebookEdit"]
  }
}
```

Defense in depth: even if `--tools ""` somehow leaks a built-in tool through, the deny list catches it.

## Launcher

The skill ships a reference copy at `~/.claude/ralph/ralph` plus supporting files in `~/.claude/ralph/`. Per-project `bin/ralph` is a **full vendored copy** of that launcher — not a shim — so the project does not depend on the skill installation at runtime.

`/ralph init` copies the launcher and its siblings into the project:

```
project/
├── bin/
│   ├── ralph                     # vendored launcher (full copy)
│   └── ralph.d/                  # vendored support files
│       ├── Dockerfile
│       ├── init-firewall.sh
│       ├── claude-settings.json
│       ├── mask-env.js
│       └── ralph-tools/          # MCP server source (no node_modules)
```

The vendored `bin/ralph` is patched to default `RALPH_HOME` to `bin/ralph.d/`:

```bash
SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd -P)"
RALPH_HOME="${RALPH_HOME:-$SCRIPT_DIR/ralph.d}"
```

`bin/ralph` and `bin/ralph.d/**` are listed in `sensitive_paths`; the loop cannot rewrite them through the MCP tool surface. Humans and the planner *can* edit them — that's intentional, so per-project customizations (e.g., a tweaked `Dockerfile`) are possible. Re-running `/ralph init` overwrites them; commit and merge any kept customizations before re-init.

### Launcher contract

```
bin/ralph <subcommand> [args]

build         build the image from this project's manifest (cached by manifest hash)
up            start the container; init firewall; mount sensitive overlays
down          stop and remove the container (volumes survive)
status        show image/container state
logs          tail container logs
shell         interactive zsh (uses a separate, more permissive image — interactive-only)
audit         dump effective runtime config; sha256-stamp the manifest
ralph [N]     run the loop up to N iterations (or unlimited if omitted), [--budget USD]
mask-env      regenerate .ralph/masked/ from real $ROOT/.env etc. (deterministic, no LLM)
progress      print IMPLEMENTATIONPLAN.md progress summary
sync          fetch commits from .ralph/sandbox-workspace back into the real workspace
test          run the ralph-tools MCP server's unit tests (host-side; auto-installs deps)
```

### Workspace strategy

At `up`:
1. Hardlink-copy `$ROOT` → `$ROOT/.ralph/sandbox-workspace/` via `cp -al` (or `cp -c` on APFS).
2. For each entry in `sensitive_paths` that's a real file in `$ROOT`, replace the hardlink in the sandbox with a real masked file from `$ROOT/.ralph/masked/<path>`. The host's real file is untouched.
3. Bind-mount the sandbox copy at `/workspace` ro/rw appropriately.
4. Bind-mount `$ROOT/ralph.config.yaml` at `/etc/ralph/config.yaml` ro.
5. Bind-mount `$ROOT/.ralph/masked/` at `/etc/ralph/masked/` ro (this is the source the MCP server reads when injecting masked env into subprocesses).

At each iteration end, the launcher runs `git fetch ./.ralph/sandbox-workspace/.git <branch>` from `$ROOT` to bring sandbox commits back into the real repo.

### Audit gate

After `build`, the launcher requires an audit pass before `ralph` can run:
1. `bin/ralph audit` computes a sha256 over the canonical YAML of `ralph.config.yaml`, writes it to `.ralph/last-audit.json` with a timestamp.
2. `bin/ralph ralph` reads `.ralph/last-audit.json`, re-computes the manifest sha256, refuses to start if mismatched ("manifest changed since audit; re-run `bin/ralph audit`").

This forces the human to re-audit any time the manifest changes, including if the loop somehow manages to tamper with it.

## Loop protocol

`PROMPT.md` for build mode. Replace `{{LINT_CMD}}` / `{{TEST_CMD}}` / `{{MODEL_NAME}}` with planner-chosen values (or just let the loop call `run_lint()` / `run_tests()` — see below).

```markdown
# Ralph build loop

You are one iteration of an autonomous build loop. Pick the single most important
unblocked task from `IMPLEMENTATIONPLAN.md`, implement it completely, commit, exit.

The loop re-invokes you for the next task. You have NO Bash, NO Edit/Write/Read
built-in tools, NO WebFetch, NO WebSearch. Everything you do flows through the
`ralph_tools` MCP server. Available verbs (use the `mcp__ralph_tools__` prefix):

**Reading:**
- `read_workspace_file(path)` — files under `/workspace`. Refuses sensitive paths.
- `git_status()`, `git_diff([path])` — current state of the working tree.

**Writing:**
- `write_workspace_file(path, content)` — overwrite a file under `/workspace`.
- `apply_patch(unified_diff)` — apply a diff (preferred for surgical edits).

**Building / testing:**
- `run_lint()` — runs the project's lint command (configured per-project).
- `run_tests([scope])` — runs the test suite (optionally scoped).
- `install_dependencies()` — runs the project's dep install command.

**Committing:**
- `git_commit(subject, body)` — auto-stages files YOU touched this iteration. Refuses
  amend/force/push. Identity comes from the manifest.

**Plan mutation:**
- `mark_complete(bullet_id)` — marks an `IMPLEMENTATIONPLAN.md` bullet `[x]`.
- `mark_blocked(bullet_id, reason)` — moves a bullet into `## Blocked` with reason.
- `mark_all_done()` — emit when no bullets remain. Loop terminates.

`bullet_id` is the kebab-case slug from the bullet's `[id:<slug>]` marker (e.g., for `- [ ] [id:add-pagination] Convert messages field…`, the id is `"add-pagination"`). If a bullet has no explicit marker, the server falls back to matching against the bullet's full normalized text — fragile, prefer explicit ids.

**Spec/research:**
- (Project-dependent — manifest may add `context7`, `postgres-readonly`, etc.)

## Step 1: Choose the task

Call `read_workspace_file("IMPLEMENTATIONPLAN.md")`. Then:

1. Collect all `- [ ]` bullets.
2. If none remain → `mcp__ralph_tools__mark_all_done()` and exit.
3. Re-evaluate blockers from scratch this iteration. Don't trust an existing
   `## Blocked` list — probe current state. Try the action that the bullet implies;
   if it fails because of a capability you don't have, then it's blocked.
4. From unblocked bullets, pick the most important — foundational utilities first,
   in-progress sections continued, top-to-bottom within a section.
5. Announce: emit a single line `CHOSEN: <bullet text>` so the loop driver can
   surface it. This is text in your assistant message, not a tool call.

## Step 2: Implement

- Read spec file(s) cited in the bullet first (`read_workspace_file("specs/foo.md")`).
- Grep before assuming not-implemented — call `git_diff` or `read_workspace_file` on
  related paths to confirm.
- If the bullet implies an external library you're unsure of, the planner should
  have pre-fetched docs into `specs/`. Read them there. The loop has no WebFetch.
- Implement only what the bullet describes. Don't drift.

## Step 3: Verify

1. Call `mcp__ralph_tools__run_lint()`. If it fails on your changes, fix and re-run.
2. Call `mcp__ralph_tools__run_tests()`. If it fails on your changes, fix and re-run.
3. If you cannot get the suite green after 3 attempts, mark the bullet blocked
   (`mark_blocked(<bullet_id>, "test suite red — <one-line cause>")`) and exit.
4. **Do not commit with a red suite. Do not mark a bullet `[x]` with a red suite.**

## Step 4: Mark and commit

1. Call `mark_complete(<bullet_id>)`. The MCP server updates the plan.
2. Call `git_commit(subject, body)`. Subject = bullet text stripped of `- [ ]` /
   citation. Body = one line citing the spec file. The server auto-stages exactly
   the files you wrote this iteration.

## Rules

- **One bullet per iteration.** Exit after the commit.
- **Plan mutation is mark_* only.** Direct writes to `IMPLEMENTATIONPLAN.md` are refused.
- **No scope creep.** Spotted a gap? It belongs in a Followups bullet; ask via
  `mark_blocked` if you need the human to add it.
- **Never touch sensitive paths.** The MCP server enforces this; don't try.
- **If quotas exhaust** (max_tool_calls, max_bytes_read, max_bytes_written), the
  current iteration terminates with `mark_blocked(<bullet>, "quota exhausted: <which>")`.
```

## Plan mode

Planning happens on the host (`/ralph plan` skill subcommand), not inside the
container. There is no `PROMPT_plan.md` shipped to the container; the planner
operates with the host `claude` session's full tool surface.

The planner's job, in summary:

1. Read existing source. Identify the stack, conventions, untrusted boundaries.
2. Write `specs/*.{md,html,json}` — pre-fetched docs, schema dumps, sample data.
   Whatever the loop will need to consume without network.
3. Write `IMPLEMENTATIONPLAN.md` — ordered bullets, each citing a spec.
4. Write `ralph.config.yaml` — the security manifest.
5. Enumerate `masked_env_vars` by grepping the source for `ENV[`, `process.env.`,
   `os.getenv`, etc. (no LLM read of real `.env` content).
6. Run `bin/ralph mask-env` to generate `.ralph/masked/` from real host files.
7. Run `bin/ralph build && bin/ralph up && bin/ralph audit > .ralph/audit.md`.
8. Read the audit, confirm it matches intent. If not, fix the manifest and re-audit.
9. Present the audit to the human for review before declaring planning complete.

The human's gate: read `.ralph/audit.md`. Sign off (any future tampering with the
manifest will fail the sha256 check at `bin/ralph ralph` time).

After approval, the human runs `bin/ralph up && bin/ralph ralph [N]`.
